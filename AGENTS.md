# Repository Guidelines

## Структура проекта и организация модулей

Это приложение на Laravel 12 и PHP 8.2. Основной код находится в `app/`, а пользовательские слои вынесены в `app/DTO`, `app/Repositories` и `app/Services`. Обработка HTTP-входа начинается в `app/Http`, маршруты лежат в `routes/`, а Blade-шаблоны, JavaScript и CSS находятся в `resources/`. Миграции, фабрики и сиды расположены в `database/`. Тесты разделены на `tests/Feature` для HTTP- и интеграционных сценариев и `tests/Unit` для изолированной логики.

## Команды сборки, тестирования и разработки

По возможности используйте скрипты Composer:

- `composer setup` устанавливает PHP- и Node-зависимости, создает `.env`, генерирует `APP_KEY`, запускает миграции и собирает фронтенд-ассеты.
- `composer dev` одновременно запускает локальный Laravel-сервер, обработчик очереди, просмотр логов и Vite dev server.
- `composer test` очищает кэш конфигурации и запускает набор тестов PHPUnit через Artisan.
- `npm run dev` запускает только Vite watcher.
- `npm run build` создает production-сборку фронтенда.

## Стиль кода и соглашения об именовании

Следуйте `.editorconfig`: UTF-8, окончания строк LF, пробелы и отступ 4 пробела. PHP-классы должны соответствовать PSR-4, а имена файлов должны совпадать с именами классов, например `PostRepository.php` и `AuthService.php`. Используйте `StudlyCase` для классов, `camelCase` для методов и переменных, а также ориентированные на таблицы имена во множественном числе для Eloquent-моделей только там, где этого требуют соглашения Laravel. Для форматирования используйте `npm run format`, который запускает `prettier` для PHP, JS, CSS, JSON, Markdown и YAML.

## Рекомендации по тестированию

Проект использует PHPUnit 11 через `php artisan test`. Добавляйте feature-тесты для маршрутов, авторизации и поведения при работе с хранилищем; unit-тесты пишите для DTO, сервисов и логики репозиториев. Файлы тестов называйте с суффиксом `*Test.php` и размещайте в соответствующей директории набора. Предпочтительны точечные проверки вместо широких дампов ответа.

## Правила коммитов и pull request

Последние коммиты используют короткие императивные заголовки в нижнем регистре, например `add rate limiter` и `remove APP_KEY from env_example`. Делайте коммиты небольшими и предметными. В pull request нужно описать изменение поведения, указать миграции или изменения окружения, приложить ссылку на связанную задачу и добавить скриншоты только если меняется UI.

## Советы по безопасности и конфигурации

Никогда не коммитьте реальные секреты в `.env`. Держите `.env.example` очищенным, особенно от `APP_KEY` и учетных данных базы данных. Перед запуском тестов проверьте `phpunit.xml` и локальные настройки БД, а конфигурацию из `docker/` используйте только если локальные PHP или PostgreSQL недоступны.
